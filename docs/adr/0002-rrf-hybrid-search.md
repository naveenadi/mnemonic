# ADR-0002: Hybrid Search via RRF Fusion (BM25 + Vector)

**Status**: Accepted  
**Date**: 2026-06-20

## Context

Knowledge base search needs both exact keyword matching (for code symbols, proper names, rare terms) and semantic similarity (for paraphrased concepts, fuzzy recall). Each backend has complementary failure modes: FTS misses semantically similar docs with different vocabulary; vector search misses exact matches when they're uncommon in the embedding space.

## Decision

Use Reciprocal Rank Fusion (RRF) to merge ranked lists from multiple sub-queries. Each search runs:

1. **One `lex` query** — BM25 via FTS5 (exact keyword match)
2. **One `vec` query** — cosine similarity via sqlite-vec (semantic match)
3. **N `vec` queries** — from LLM query expansion (alternative phrasings)
4. **One `hyde` query** — vector search against a hypothetical document generated by the LLM

Fusion: each sub-query contributes `weight / (K + rank)` where K=60. Original queries get 2x weight vs. expanded ones. Top-rank bonus adds +0.05 for rank-1 docs, +0.02 for ranks 2-3.

After fusion, an optional LLM reranking step blends RRF scores with reranker scores using position-aware weights (75/25 for top 3, 60/40 for 4-10, 40/60 for rest).

## Consequences

**Positive**
- Best of both worlds: exact matches don't get drowned by semantic scores
- RRF is parameter-light compared to learned fusion (one constant K)
- Top-rank bonus prevents the "many weak signals vs one strong signal" problem
- Each sub-query is independent and can be run in parallel

**Negative**
- RRF is uncalibrated — scores are relative, not absolute relevance measures
- Fusion cost scales with number of sub-queries × candidates per query
- Position-aware reranking weights are heuristic (tuned by intuition, not data)
